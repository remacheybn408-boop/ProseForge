# 行为规范：AI 长篇小说连续写作工程流程

> 适用项目：100 万字长篇小说 | 执行对象：AI Agent (Hermes/Claude/GPT)
> 版本：分阶段执行合并版 V3

---

## 零、核心原则

这不是写作建议。这是**工程流程规范**。

AI Agent 写长篇小说会犯三种系统病：

1. **上下文断裂**：每章像在真空中写，只看大纲不读上一章结尾
2. **SQLite 忘用**：凭记忆写，越往后丢设定越多
3. **字数缩水**：从"创作模式"滑入"任务完成模式"，逐章缩水

**解决方式不是"我会注意"，而是硬性门禁。**

```
SQLite 负责记住 → 门禁负责不偷懒 → 摘要负责不迷路 → 版本负责能回滚
```

## 一、总原则

1. 不读上一章结尾，禁止写下一章
2. 不查 SQLite，禁止写正文
3. 不生成 context_pack，禁止写正文
4. 不生成 chapter_task_card，禁止写正文
5. 不满足字数门禁，禁止交稿
6. 不做连续性检查，禁止进入下一章
7. 不做场景质量检查，禁止进入下一章
8. 不做反 AI 味检查，禁止进入下一章
9. 每章完成后必须入库
10. 每章完成后必须保存版本
11. 每 3 章做一次阶段复盘
12. 任何时候不能只靠大纲写章节
13. 任何时候不能直接覆盖原稿

**一句话：先读库，再写作，再检查，再入库，再复盘。**

## 二、每章强制流程（8 步）

```
pre → task_card → write → word_count → continuity → scene → anti_ai → ingest
```

### Step 1: pre — 写作前准备

**必须完成：**

1. 如果不是第 1 章，读取上一章全文
2. 提取上一章最后 800-1200 字
3. 读取最近 3 章摘要
4. 检索人物
5. 检索世界观
6. 检索伏笔
7. 检索 reader_promises
8. 检索 writing_rules
9. 检索相关章节切片
10. 生成 context_pack
11. 写入 novel_logs
12. 保存 pipeline_state.json（allowed_to_write = true）

**pre 失败则停止写作。**

### Step 2: task_card — 章节任务卡

必须包含 22 个字段：章节编号、标题、目标字数、字数类型、上一章结尾状态、本章开头承接细节、主要剧情目标、必须推进的伏笔、必须推进/兑现的 reader_promises、≥4 个场景设计、结尾钩子、禁止事项。

**无任务卡则禁止写正文。**

### Step 3: write — 正文写作

- 普通章：初稿目标 3500-3800 字
- ≥4 个有效场景块
- 每场景块：动作 + 对话 + 感官 + 冲突 + 信息变化
- 禁止：硬科普代替剧情、总结腔代替现场描写、为凑字写空泛心理

### Step 4: word_count — 字数门禁

| 字数 | 判定 |
|------|------|
| 3300-4200 | ✓ 正常通过 |
| 4200-5000 | ⚠ 允许但检查是否拖沓 |
| 3000-3299 | ⚠ 黄灯：须场景+连续性+摘要均通过 |
| < 3000 | ⛔ 红灯：必须重写/扩写 |

**补充规则：**
- 连续两章 < 3300 → 暂停，检查缩水模式
- 连续三章 < 3500 → 启动阶段复盘
- 版本 ≥ 3 且黄灯 → 拦截（疑似 patch 凑数），要求重铺场景

**扩写方向（允许）：**
场景动作、对话冲突、环境压力、实验过程、失败代价、伏笔推进、结尾钩子

**禁止凑字方式：**
空泛心理、重复设定、大段科普、无意义对话、水天气、流水账

### Step 5: continuity — 连续性检查

检查本章开头是否承接上一章结尾：
1. 上章最后的人/物/伤势/药/异常能力，本章是否处理
2. 时间是否顺畅
3. 地点是否合理
4. 人物称呼是否一致
5. 男主能力是否突然变强
6. 物理学设定是否违背前文
7. 伏笔是否遗漏/重复/提前回收

结果写入 `continuity_checks` 表。

### Step 6: scene — 场景质量门禁

≥4 个有效场景。每个有效场景须回答：
- 在哪里？谁在场？冲突是什么？
- 角色想要什么？阻碍是什么？
- 场景结束时发生了什么变化？
- 是否有动作/对话/感官/心理至少三种？
- 是否推进主线/人物/伏笔/世界观之一？

**无效场景：** 只有解释无行动、只有心理无外部变化、只有设定无冲突、只有对话无目标、可删掉不影响剧情。

### Step 7: anti_ai — 反 AI 腔门禁

**10 项检测：**
1. 总结腔（"他意识到""这意味着"过多）
2. 解释腔（连续多段解释设定）
3. 论文式对话
4. 过度工整（所有句子节奏一致）
5. 禁止句式（"不是A而是B""那一刻她终于明白""她从未想过"）
6. 缺少动作描写
7. 缺少口语化表达
8. 缺少有记忆点的细节
9. 物理学说明像硬科普
10. 像大纲扩写而非现场发生

**修复方式：** 用动作替代心理总结、冲突对话替代设定说明、失败/误差/代价表现科学探索、具体物品承载情绪。

### Step 8: ingest — 自动化入库

自动完成：
1. 保存 chapters（正文 + 标题 + 字数）
2. 保存 chapter_versions（版本快照）
3. 自动切片 chapter_chunks（800-1500 字/片）
4. 生成 chapter_summaries（短摘要 + 长摘要 + key_events）
5. 同步 FTS 全文索引
6. 更新 novels 字数统计
7. 写入 novel_logs

**ingest 失败则章节视为未完成，不允许进入下一章。**

## 三、阶段性复盘

### 每 3 章阶段复盘

检查：连续性、字数趋势、SQLite 使用、人物状态、伏笔推进、reader_promises 状态、AI 腔趋势

### 每 10 章剧情节奏复盘（第二阶段启用）

主线推进、男主成长、爽点兑现、新坑追踪、人物冷热、设定复杂度、水文倾向、偏离主题检查

## 四、上下文承接规范

每章开头必须回答 13 个问题：
1. 上一章最后动作是什么？
2. 上一章最后情绪是什么？
3. 上一章未解决问题是什么？
4. 上一章最后重要物品是什么？
5. 上一章最后重要人物是谁？
6. 男主身体状态如何？
7. 男主是否受伤/疲惫/中毒/突破？
8. 男主心理状态如何？
9. 男主修为/能力状态？
10. 当前时间？
11. 当前地点？
12. 重要物品在哪里？
13. 本章第一场景如何承接以上？

## 五、执行闭环

### Pipeline State 文件锁

`pre` 完成后生成 `chapter_XXX_state.json`，`post` 前验证其存在且 `allowed_to_write = true`。

### 禁止 Patch 凑数检测

黄灯 + 版本 ≥ 3 → 拦截，要求回到 task_card 重铺缺失场景，而非继续 patch。

### 每章验收清单

1. pre 是否完成 ✓
2. context_pack 路径 ✓
3. task_card 路径 ✓
4. 字数是否达标 ✓
5. continuity_gate 是否通过 ✓
6. scene_quality_gate 是否通过 ✓
7. anti_ai_style_gate 是否通过 ✓
8. ingest 是否完成 ✓
9. reader_promises 更新情况 ✓
10. chapter_versions 已保存 ✓
11. novel_logs 已记录 ✓

## 六、分阶段落地策略

### 第一阶段（1-10 章）

- 8 步精简流水线
- 2 张新表：chapter_versions, reader_promises
- 6 道门禁
- 暂不做：character_state_snapshots, timeline_events, location_states, chapter_report, 10 章复盘

### 第二阶段（10 章后）

- 新增 4 张状态表
- chapter_report 正式报告
- 10 章复盘 + 30 章分卷复盘

## 七、不做什么

- ❌ 18 个独立脚本（全部合并到 chapter_pipeline.py）
- ❌ 36 步重型流水线
- ❌ Web UI / FastAPI / Docker / Nginx
- ❌ 向量数据库
- ❌ 自动写 100 万字
- ❌ 每章强制 4000+ 字
- ❌ 直接覆盖原稿
- ❌ 只靠大纲写章节
